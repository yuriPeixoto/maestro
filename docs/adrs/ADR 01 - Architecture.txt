-------------------------------------------------------------------------------

1. DECISÃO ARQUITETURAL: AGENTE DE INGESTÃO (WORKER) EM GO
---------------------------------------------------------
- ESCOLHA: Implementação de binários leves em Go para coleta local[cite: 45, 62].
- MOTIVAÇÃO: Performance bruta e baixo consumo de memória (CPU/RAM) para não 
  impactar o hardware do cliente/servidor monitorado[cite: 45].
- PADRÃO TÉCNICO: Worker Pool com Buffered Channels e Goroutines[cite: 83, 84].
  * Canais (channels) limitam o buffer para evitar travamentos na aplicação[cite: 85].
  * Batching: Envio de eventos em lotes (ex: 50 itens ou 5 seg) para reduzir 
    overhead de conexão em 90%[cite: 64, 65, 86].

2. DECISÃO DE STORAGE: CLICKHOUSE (COLUMNAR DB)
-----------------------------------------------
- ESCOLHA: Clickhouse como banco de dados principal de telemetria[cite: 46].
- MOTIVAÇÃO: Alta performance em ingestão e consultas analíticas (OLAP). 
  Diferencial técnico para perfil sênior[cite: 46, 47].
- ESTRATÉGIA DE TABELA:
  * Engine: MergeTree para alta performance[cite: 58, 83].
  * Otimização: Uso de LowCardinality para campos repetitivos (host_id, tenant_id) 
    para acelerar queries e reduzir consumo de memória[cite: 78, 81, 82].
  * Compressão: Uso de codec(ZSTD(1)) para logs de texto, reduzindo ocupação 
    em disco em até 80%[cite: 80].
  * Particionamento: Por dia ou mês (toYYYYMM), otimizando leituras de 
    "últimas 24h"[cite: 60, 61, 81].

3. DECISÃO DE STREAMING E DESACOPLAMENTO: REDIS STREAMS
-------------------------------------------------------
- ESCOLHA: Redis Streams entre a API de Ingestão e o Banco de Dados[cite: 72, 76].
- MOTIVAÇÃO: Baixa sobrecarga de infraestrutura comparado ao Kafka[cite: 46]. 
  Garante que se o banco (ClickHouse) oscilar, os logs fiquem enfileirados 
  sem perda de dados[cite: 76, 77].

4. DECISÃO DE INTELIGÊNCIA E API: FASTAPI (PYTHON)
--------------------------------------------------
- ESCOLHA: Backend de controle e dashboard em Python/FastAPI[cite: 47, 48].
- MOTIVAÇÃO: Velocidade de desenvolvimento e ecossistema rico para futuras 
  implementações de análise preditiva e processamento de dados[cite: 48, 88].
- ALERT ENGINE: Sistema desacoplado (Worker independente) para checagem de 
  thresholds em paralelo, sem afetar a UI[cite: 50, 51, 79].

5. CONTRATO DE DADOS (JSON POLIMÓRFICO)
---------------------------------------
- ESTRUTURA: Um único contrato para métricas e logs[cite: 66, 67].
- CAMPOS BASE: timestamp, host_id, tenant_id, type, payload e tags[cite: 68, 81].
- OBJETIVO: Unificação da telemetria em uma única tabela para facilitar a 
  correlação de eventos (ex: erro no log vs. pico de CPU no mesmo instante)[cite: 67].

6. CANAIS DE NOTIFICAÇÃO (ALERTING)
-----------------------------------
- Nível 1 (Operacional): Webhooks para Discord/Slack (Imediato)[cite: 53, 54].
- Nível 2 (Sênior): Telegram Bot API (Canal de suporte oficial/privado)[cite: 54, 55].
- Nível 3 (Crítico): SMTP/Email para falhas de infraestrutura base[cite: 56, 57].

-------------------------------------------------------------------------------
NOTAS FINAIS:
O foco desta arquitetura é deslocar o "trabalho sujo" (coleta pesada) para Go 
e manter a "inteligência" (dashboards e alertas) em Python[cite: 87, 88].
